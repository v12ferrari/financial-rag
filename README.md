# 금융 공시 분석 RAG 시스템

DART 공시 문서를 로컬 LLM으로 분석하는 온프레미스 RAG 시스템.
외부 API 없이 전 과정을 로컬에서 처리한다.

## 아키텍처
'''
PDF 공시
│
├─ 벡터 검색 (Chroma + nomic-embed-text) ── 의미 기반
│                                              │
├─ BM25 검색 (rank_bm25) ─────────────────── 키워드 기반
│                                              │
└────────── EnsembleRetriever (5:5) ───────────┘
│
llama3.3:70b (temperature=0)
│
답변 + 출처 페이지 인용
'''




## 스택

| 계층 | 기술 |
|------|------|
| LLM | Ollama · llama3.3:70b (Q4, ~40GB) |
| 임베딩 | nomic-embed-text |
| 벡터DB | ChromaDB (컬렉션 단위 격리) |
| 검색 | BM25 + 벡터 하이브리드 |
| 오케스트레이션 | LangChain (LCEL) |
| UI | Streamlit |

전 구성요소가 로컬에서 동작하며, 외부 네트워크 호출이 없다.

## 문제 해결 기록

초기 구현은 정량 데이터 추출에 실패했다. 원인을 단계별로 분리해 진단했다.

**1. 벡터 검색이 재무제표 표를 찾지 못함**

임베딩 검색은 "영업이익"으로 질의하면 회계정책 *설명 문단*만 반환하고,
정작 숫자가 담긴 표는 놓쳤다. 숫자 나열은 의미 벡터 공간에서 위치가
불안정하기 때문이다. 청크 크기를 500→800→1500→3000으로 늘려봤으나
해결되지 않았다.

→ **BM25 하이브리드 검색 도입.** 키워드 정확 매칭이 표를 안정적으로
   회수하면서 해결. 임베딩의 구조적 약점을 보완하는 접근이다.

**2. 동일 컨텍스트에서 8B 모델은 표 해석 실패**

검색이 표를 찾아준 뒤에도 8B는 "Not found"를 반환했다. 표는 레이블 없는
숫자 나열(`영업이익 2,967,682 53,663,274 ... 57,232,797`)로 들어가므로,
열 구조를 추론해야 정답을 특정할 수 있다.

| 모델 | 동일 컨텍스트 결과 |
|------|-------------------|
| llama3.1:8b | ❌ Not found |
| llama3.3:70b | ✅ 57,232,797 |

→ 정량 추출 태스크에서는 모델 규모가 유의미하게 작용함을 실측 확인.

**3. 문서 간 데이터 오염**

초기 구현에서 SK하이닉스 질의에 삼성전자 수치가 반환됐다. 동일 ChromaDB
컬렉션에 두 문서를 적재한 것이 원인.

→ `collection_name` 을 필수 인자로 강제해 구조적으로 차단.

**4. 질의당 PDF 재파싱**

매 질의마다 323페이지 PDF를 다시 로드해 90초 이상 소요됐다.

→ `lru_cache` 적용. **99.6초 → 3.3초.**

## 성능

| 항목 | 값 |
|------|-----|
| 추론 속도 (70B Q4, M4 Pro 48GB) | 5.67 tok/s |
| 질의 응답 (캐시 후) | 3~7초 |
| 초기 인덱싱 | 323p → 330 chunks |

## 실행

```bash
# 모델 준비
ollama pull llama3.3:70b
ollama pull nomic-embed-text

# 서버 (KV 캐시 양자화로 메모리 절감)
OLLAMA_FLASH_ATTENTION=1 OLLAMA_KV_CACHE_TYPE=q8_0 \
OLLAMA_CONTEXT_LENGTH=16384 ollama serve

# 의존성
pip install -r requirements.txt

# 인덱싱
python -c "from src.vectorstore import build_store; from src.config import DATA_DIR; \
build_store('samsung', DATA_DIR / '삼성전자1분기.pdf')"

# 데모
streamlit run app.py
```

## 구조
'''
src/
├── config.py       # 모델·경로·RAG 파라미터
├── vectorstore.py  # 로드 → 청킹 → 임베딩 → 저장
├── retriever.py    # BM25 + 벡터 하이브리드
├── chain.py        # RAG 체인 + 출처 인용
└── compare.py      # 다중 문서 비교
app.py              # Streamlit UI
'''

## 설계 의도

- **온프레미스**: 금융·보안 도메인은 외부 API로 문서를 전송할 수 없는
  경우가 많다. 전 파이프라인을 로컬로 구성해 데이터가 기기를 벗어나지 않는다.
- **출처 인용**: 모든 답변에 근거 페이지를 함께 반환한다. LLM 환각을
  사람이 검증할 수 있어야 실무에 쓸 수 있다.
- **temperature=0**: 사실 추출 태스크에 생성 다양성은 불필요하며 환각만 늘린다.

## 감성분석 성능

고정 평가셋(n=15, 수동 라벨링) 기준.

| 모델 | 프롬프트 | exact | direction | MAE |
|------|---------|-------|-----------|-----|
| llama3.1:8b | baseline | 53% | 60% | 0.53 |
| llama3.1:8b | + 도메인 규칙 | 60% | 67% | 0.47 |
| llama3.3:70b | baseline | 73% | 80% | 0.33 |
| llama3.3:70b | + 도메인 규칙 | **87%** | **93%** | **0.20** |

도메인 규칙 주입 효과가 8B(+7%p)보다 70B(+14%p)에서 두 배로 크게 나타났다.
지시 이해·적용 능력이 모델 규모에 의존함을 시사한다.
